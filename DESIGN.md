# allisp 設計決定書

日付: 2026-07-15
更新: 2026-07-20
状態: v1 実装済み

allisp は、思考・分析・業務手順などを S 式で表現した `.lisp` ファイルを
実行するための方言とその処理系である。特定のノートリポジトリやディレクトリ構成には依存しない。
定義があるものは決定論的に評価し、定義がないものは LLM がLispコードへ
loweringする。生成コードは全体が解決済みの場合だけ決定論的評価器が実行する。

## 決定事項(依存順)

### 1. 一次出力 = 評価済みS式 + トレース
- `foo.lisp` を実行すると `foo.result.lisp`(全トップレベル式の評価結果)と `foo.trace.lisp`(LLM 呼び出しの完全な記録)を生成する。
- 結果が言語内(S式)に留まるため、前の式の評価結果を次の式の入力にする合成が可能。
- 自然言語レポートは result からのレンダリング層として将来後付けする(v2)。

### 2. 新方言 allisp、Common Lisp に近い S 式構文
- 独自の評価器を持つ方言を定義する。`defun` / `defmacro` などの基本語彙と、
  `→` を含む Unicode シンボルを扱える。
- Common Lisp そのものとの 100% 互換は保証しない。言語差分は
  [docs/language.md](docs/language.md) でこのリポジトリ内に公開する。

### 3. 実行境界 = 未束縛シンボル → LLMコード生成
- 定義済みの関数・マクロ・算術は決定論的に計算する。
- 未束縛の関数適用はエラーにせず、式+文脈を LLM に渡してLispコードを受け取る「オラクル呼び出し」にする。
- 生成コードは全演算が解決済みの場合だけ、LLMフォールバックを禁止して評価する。
- `(llm ...)` で強制的に LLM へ、`(pure ...)` でフォールバック禁止(未定義はエラー)。
- 全 LLM 呼び出しはトレースに記録する。

### 4. 永続オラクルキャッシュ
- キー = (式, 評価文脈, モデルID, プロンプト版) のハッシュ。値 = LLM が生成したLispコード。
- 保存先: プロジェクトルートの `.allisp/oracle/`。git 管理するかは選択制。
- プロジェクトルートは、入力ファイルから上方向に `.allisp/` または `.git/` を持つ
  最寄りのディレクトリ。見つからなければ入力ファイルのディレクトリを使う。
- 再実行はキャッシュからの決定論的リプレイ。`--refresh` / `(llm :fresh t)` で意図的に再思考させる。

### 5. オラクル文脈 = 参照関係の自動同梱
- 対象式 + その式が参照するシンボルの束縛値 + 該当マクロ/関数定義
  (docstring 含む、`@use` 先も追跡) + 囲むトップレベルフォームを、評価器が
  依存解析して自動同梱する。
- ファイル全体は渡さない(キャッシュ安定性とコストのため)。`(llm :context :file)` で明示的に拡大可能。

### 6. オラクル出力契約 = Lispコード1式のみ
- LLM は実行可能なLispコードを一式だけ返す。一意に定まらなければ
  `(intermediate-code ...)` を返す。confidence 等のメタデータは契約に含めない。
- 生の API 応答はトレース側に残す。パース不能なら N 回再試行 → エラー値。

### 7. ホスト言語 = Common Lisp (SBCL)
- Roswell 管理の SBCL 2.5.x(確認済み)。ASDF システム + Roswell スクリプトで構成。
- CL reader は `→` も dotted pair もデータとして読めるため reader 自作の負担が小さい。

### 8. LLM 経路 = claude CLI サブプロセス
- `uiop:run-program` で認証済みの `claude -p --model <model>` を呼ぶ。
- 認証情報は claude CLI 側で管理し、allisp のソースや設定には保持しない。
- バックエンドは protocol で抽象化し、将来 Anthropic API 直接呼び出しに差し替え可能にする。

### 9. 評価器 = メタ循環評価器を自作
- CL の reader でS式を読み、自前の環境構造と eval 関数で評価する。
- defmacro も自前実装(CL と同型の意味論)。CL の eval / コンパイラには乗せない。
- 理由: 未束縛検出・トレース・依存解析・`pure` の境界制御を第一級で持つため。

### 10. ビルトイン = 特殊形式自作 + CL 白名単パススルー
- quote / if / let / let* / lambda / defun / defmacro / quasiquote / cond / progn / and / or 等の特殊形式は自作評価器に実装。
- 一般関数(算術・比較・リスト・文字列・assoc 等)は安全な CL 関数の白名単を env に束縛して流用。
- 白名単外は確実に「未束縛 → LLM」に落ちる。白名単は随時拡張可能。

### 11. 名前空間 = ファイル=モジュール、@use で継承
- 各ファイルは新しい環境で先頭から逐次評価される。
- `(@use "path")` は参照先ファイルを評価し、その全定義を現在環境に取り込む。冪等(重複読み込みは1回)。
- パスは呼び出し元ファイルからの相対パスとし、特定のディレクトリ名は要求しない。

### 12. 成果物配置
- result / trace: 対象ファイルと同じディレクトリの `output/` に生成する。
- オラクルキャッシュ: プロジェクトルートの `.allisp/oracle/` に集中
  (ハッシュキーなのでファイル横断共有可)。

### 13. エラー = エラー値化 + 継続実行
- 失敗した式は再試行 N 回の後 `(error :type … :form … :detail …)` という第一級のエラー値になり、result に埋まって残りの式の評価は続行(部分評価)。
- エラーが1つでもあれば exit code 非ゼロ。`--strict` で最初のエラーで即停止。
- 成功部分はキャッシュ済みなので、再実行は失敗箇所だけ再問い合わせになる。

### 14. デフォルトモデル = sonnet
- 式に `(llm :model …)` の指定があればそれを最優先。実行単位の `--model` でも上書き可能。
- 高い推論能力が必要な式は明示的に opus を指定する運用。
- モデル ID はキャッシュキーに含まれる。

### 15. CLI エントリポイント
- ファイル実行: `allisp run <file>`。result / trace を入力ファイル側の `output/` に生成する。
- 単発実行: `allisp --one-liner "<form>..."`。文字列内の各フォームを順番に評価し、
  最後の値を標準出力へ S 式で表示する。result / trace ファイルは生成しない。
- 共通オプション: `--refresh` / `--strict` / `--model` / `--dry-run`(LLM 行き箇所の予告)。
- 受け入れ基準:
  1. 直後の再実行が全キャッシュヒットで決定論的に完走する。
  2. `make test` は外部の allisp ソースファイルなしで完走する。
- REPL / watch / 自然言語レンダリング / MCP サーバー化は v2 以降。

### 16. 評価値のコード生成 = `generate-file`
- `(generate-file "path" body...)` は body の最終評価値を別ファイルの S 式として書き出す。
- この公開構文は組み込みマクロとし、実際の書き出しは内部特殊形式が担う。
- 生成先に `generated-by` マクロ定義とマーカーを挿入し、生成元ファイル、
  元フォーム、生成時刻を `*generated-by*` で参照可能にする。
- dry-run では書き出さず、生成予定パスのみ報告する。呼び出し元自身の上書きは禁止する。
- 書き出し先が `.lisp` 以外なら値は文字列に限り、生テキストとしてそのまま書く(2026-07-16 追加)。
  抽象 DSL → Python/シェル等の具体コードへの lowering の終端。ヘッダは埋め込まず
  (JSON 等コメント不可形式のため)、生成記録は trace 側に残す。文字列以外はエラー値。
  body 評価中のオラクル呼び出しには「Lisp文字列リテラルを一つ生成する」ルールを追加する
  (プロンプトが変わるため該当呼び出しのみ別キャッシュキーになる。既存キャッシュは無傷)。

### 17. result リプレイ = chain の媒体(2026-07-16 追加)
- `output/foo.result.lisp` は `@use` の対象にできる。`result` を組み込みフォームにし、
  `(result :v 2 :n K :form F :value V)` の評価は F を再評価せず V をデータとして扱う。
  リプレイはオラクル呼び出しゼロ。
- F が def 系(def / defvar / defparameter / define の変数形)なら名前 → V を再束縛する。
  下流ファイルは `(@use ".../foo.result.lisp")` の後、上流で def した名前をそのまま参照できる。
  名前参照のみとし、番号アクセサは作らない(上流の式の挿入で位置がずれ、黙って壊れるため)。
- defun / defmacro は復元しない(クロージャは externalize でプレースホルダになるため)。
  関数も継承したい場合は元ソースを `@use` する。
- 直近の result の値は `last-result` で参照できる。
- result 形式バージョン `:v 2` を各フォームに記録する。`:v` なしの旧形式は def の値を
  記録していない(:value が名前シンボル)ため、名前復元をスキップして誤束縛を防ぐ。
- `result` は予約語となる。

### 18. オラクルのエージェント化(2026-07-16 追加)
- 背景: 文脈同梱(決定5)は環境内の束縛しか届かず、思考 DSL が `basis` / 相対パスで指す
  外部文脈を読めないため、大きな宣言的フォームのloweringが入力の言い換えに退化していた。
- オラクルの claude CLI サブプロセスに読み取り専用ツール(Read / Glob / Grep)を
  `--allowedTools` で許可し、`--strict-mcp-config` で MCP を遮断、作業ディレクトリ =
  プロジェクトルートで起動する。書き込み系ツールは許可リスト外なので -p モードで自動拒否される。
- プロンプトに Environment 節(source file / project root / 探索指示)と
  アンチエコー規則(値はフォームの実行結果であり、再掲・言い換えではない)を追加。
  プロンプト版は `20260716-1`。現在日時はプロンプトに入れない(キャッシュキーが毎日変わるため)。
- 無効化は CLI `--no-explore`(backend 単位)。探索の有無でプロンプトが変わるため
  キャッシュキーは自然に分かれる。one-liner はソースファイルがないので Environment 節なし。
- 決定論の意味: fresh 呼び出しは呼び出し時点のリポジトリ状態を反映する。
  キャッシュリプレイは従来どおり完全決定論。再思考は `--refresh` / `(llm :fresh t)`。
- オラクルが読んだファイル一覧の trace 記録(`--output-format json` のパース)は v2 送り。

### 19. オラクル = 段階的コード生成、実行 = 決定論的評価器（2026-07-18 変更）

- 決定3・4・6の「LLMがフォームを擬似実行して値を返す」という部分を置き換える。
- LLMは未束縛フォームを実行せず、その意図を一つのLisp式へloweringする。メモリ確保、ファイル変更、送信、デプロイなどが成功したと主張してはならない。
- 生成コードに現れるすべての演算子と参照値をallispの決定論的評価器が解決できる場合だけ、`pure`相当のLLM禁止状態で評価する。これにより実行主体は常にLisp処理系になる。
- 裸のリスト値は実行可能コードではない。データを生成する場合、LLMは `(quote (...))` または `(list ...)` を生成する。
- 一意なコードを生成できない場合、LLMは `(intermediate-code :source ... :reason (:why ... :how ...) :constraints ... :candidates ...)` を生成する。`:why`は停止理由、`:how`は不足している定義・制約・判断と次のlowering手順を示す。評価器も未解決演算子を含む生成コードを同形式で包み、旧形式を正規化して実行しない。
- `intermediate-code` は不活性な特殊形式であり、そのままresultやchainに保持できる。追加の文脈が揃った後に `(llm <intermediate-code>)` と明示すると次のlowering段階へ進む。
- 永続キャッシュは評価値ではなく生成コードを`:code`に保存する。ヒット時にも現在の決定論的環境でコードを再評価する。
- traceは`:code`、`:status :executed | :intermediate`、`:value`を分離して記録する。
- プロンプト版を`20260718-1`へ更新し、旧来の値キャッシュとはキーを分離する。
- `:int`の具体loweringとして、32/64bit specialized arrayと初期化bitmapを持つ
  `allocate-memory-block`を決定論的builtinにする。未初期化要素は
  `memory-block-write`前に`memory-block-read`できず、externalizeもcontentsを読まない。
  resourceの寿命はcurrent processに限定する。

### 20. アンチ散文規則 = 計算可能な主張は文字列に埋めない(2026-07-18 追加)
- 背景: 決定19後も、オラクルが導出値(例: 成長予測の数値)を生成コードの文字列リテラル内で
  暗算して埋め込み、評価器が `(quote <散文>)` を実行するだけの退化が起きていた。
- オラクル規則に追加: 束縛から導出できる数値・件数・比較・予測・集計は文字列に書かず、
  束縛を参照する評価可能な部分式として生成する(例: `(* peak (expt (+ 1 growth) 4))`)。
  文字列は計算不能な判断にのみ使い、生成コードが計算する値を散文で再掲しない。
- これにより導出値は検算可能になり、前提の変更が値に追従する。
  プロンプト版は `20260718-2`。

### 21. result 差分 = `allisp diff`(2026-07-18 追加)
- `allisp diff <old.result.lisp> <new.result.lisp>` は 2 つの result ファイルを比較し、
  どの前提の変更がどの結論を変えたかを表示する感度分析コマンド。LLM 呼び出しなし。
- 対応付け: def 系(def / defvar / defparameter / define / defun / defmacro)は定義名で、
  それ以外はフォーム自身で照合する(前提編集はフォーム自体が変わるため名前照合が必要)。
  同一フォームの重複は出現順で照合する。
- 出力は 1 差分 = 1 S 式: `(changed :name|:form … :old … :new …)` /
  `(added …)` / `(removed …)`。ヘッダに changed / added / removed / unchanged の件数。
- exit code: 同一 = 0、差分あり = 1(diff(1) と同じ規約)。

### 22. `fix` = intermediate-code の自動解決(2026-07-18 追加)
- `(fix <form> [:rounds N] [:model M] [:fresh F])` は `<form>` を評価し、値が
  intermediate-code のとき Fix モードで再 lowering する(既定 2 巡)。
- Fix モードのオラクルは `:reason` の `:how` を指示として扱い、不足する前提・定義・選択を
  最も妥当な既定値で埋めてよい。`:candidates` があれば優先し、`:constraints` を尊重する。
  ただし導入した仮定はすべて生成コード内に明示的に束縛する(let 等)。仮定を散文に隠さない。
- 実行に至った巡の値は `(fixed :source <元フォーム> :code <生成コード> :value <評価値>)`。
  何を仮定したかが result 上で検査可能になる。修正不要の値は素通しし、
  全巡未解決なら最後の intermediate-code を返す。
- Fix モードはプロンプトに専用節を加えるため、通常 lowering とはキャッシュキーが分かれる。
  生成コード内の `fix` は `llm` と同様に未解決扱い。`fix` は予約語。
- オラクル呼び出しは通常の trace / キャッシュ機構に乗る(リプレイは決定論)。

### 23. `markdown->lisp` = markdown 文書のプログラム化(2026-07-18 追加)
- `(markdown->lisp <source> [:from :file|:text] [:out <path.lisp>] [:model M] [:fresh F] [:eval E])`
  は markdown で書かれたプロンプト・指示をオラクルに変換させ、allisp フォーム列として返す特殊形式。
- 入力元は呼び出し元ファイル基準のパス(`:from :file`、既定)または文字列そのもの(`:from :text`)。
  `:from "doc.md"` のようにパス文字列を直接渡す寛容記法も受理する(位置引数の `<source>` は不要)。
  入力元と出力先の評価は pure 相当(未束縛はオラクルに送らず即エラー値。無駄な LLM 呼び出しと
  分かりにくい二次エラーを防ぐ)。
  出力先 `:out` は `.lisp` に限り、`generate-file` と同じ `generated-by` マーカー付きで書き出す。
- 変換オラクルには散文禁止規則(決定20の拡張)を課す: 文書の言い換え文字列を禁止し、
  既存の構文・型を最優先、捉えられないドメイン構造には小さな宣言的 DSL を設計させる。
  応答契約は「フォーム列を包んだ一つの `(progn ...)`」。
- 生成コードは実行せずプログラムとして保持する(`oracle-eval` の `:execute nil` 経路、
  trace の `:status` は `:generated`)。`:eval t` のときだけ `@use` と同様に現在環境で逐次評価する。
- 通常のキャッシュ・trace・dry-run・リトライ機構に乗る。文書全文がプロンプトに含まれるため、
  文書の改訂は自然に別キャッシュキーになる。一意に変換できない文書は `intermediate-code` になり、
  ファイルは書き出さない。`markdown->lisp` は予約語。

### 24. 売りの融合 = 仕様の正本としての allisp(2026-07-19 追加)
- ユーザー指示により、mizchi 氏のツイート(2026-07-16、status 2077677183783551034)の構図
  「正本を自然言語で書くべきではなく形式的に書き、それを常に真として可読ドキュメントと
  テストオラクルを生成させる。実装はそれをパスするように書く。仕様の追加/問い合わせは
  LLM を通して形式的なモデルを介して行う」を README の最初の売りに融合した。
- allisp への対応付け: 正本 = allisp ファイルの `def` 束縛(不変条件・実例)。
  可読ドキュメント・テストオラクル(pytest)・実装(Python 等)はすべて
  `generate-file` + オラクル lowering で正本から生成する派生物。コード生成は LLM に
  委ね、実行は常に決定論的評価器(Lisp)か外部ツール(pytest)が担う(決定19と整合)。
- 「常に真」の担保 = 永続キャッシュ: 正本が不変なら全派生物はキャッシュリプレイで
  バイト同一に再生成($0)。1 条項の編集は参照するフォームだけを再生成する(決定4・5)。
- 仕様問い合わせはオラクル経由で形式モデルに対して行う。未規定の角を突く質問には
  値をでっち上げず `intermediate-code` が返り、`:reason` が矛盾する条項を、`:how` が
  追加すべき `:examples` を指す(決定19の帰結。実測で確認済み)。
- 実証: `sample/12-spec-as-source.lisp`(slugify 仕様 → doc + pytest + 実装、
  pytest 5 passed、再実行 4/4 キャッシュヒット)。評価器・プロンプトの変更はなし。

### 25. `allisp run <dir>` = ディレクトリ内 `*.lisp` の一括実行(2026-07-19 追加)
- `allisp run` の第2引数がディレクトリなら、そのディレクトリ**直下**(非再帰)の
  `*.lisp` をファイル名の辞書順に1本ずつ独立実行する。各ファイルは独自の
  `make-global-env` を持ち、束縛は他ファイルに漏れない(決定11の namespace=file と整合)。
  サブディレクトリは対象外(`output/` に `.result.lisp` / `.trace.lisp` に加え
  `generate-file` の生テキスト生成物(`.lisp` 拡張含む)が置かれるため、再帰すると
  生成物をソースとして誤実行する事故が構造的に起こる)。
- ファイル名が `.result.lisp` / `.trace.lisp` で終わるものは、直下に置かれていても
  ソース候補から除外する(`--out-dir` の事故的な指定ミス等への安全網)。
- 実行順はファイル名の辞書順(`sample/` の `01-`〜`12-` という番号接頭辞の命名慣習と一致)。
- エラー時の継続: `--strict` なしでは1ファイルの失敗(フォームエラー・パース失敗とも)は
  記録して次のファイルに進む。`--strict` 指定時は最初の失敗でバッチ全体を打ち切り、
  残りのファイルは実行しない(「最初のエラーで止まる」という `--strict` の意図を
  ファイル単位からバッチ単位へ一貫して拡張する)。
- パース段階の失敗(S式構文エラー等)も上記と同じ継続/中断ルールに統合する。
  単一ファイル実行時は非捕捉のまま CLI 最上位まで伝播して exit 3 になるが、
  ディレクトリモードではファイル単位の失敗として捕捉し、他ファイルの実行を妨げない。
- プロセス全体の終了コード: 1ファイルでも失敗があれば 1、全件成功なら 0。
- `--out-dir` はディレクトリモードでも有効で、全ファイル共通の出力先になる
  (`<out-dir>/<name>.result.lisp` / `<out-dir>/<name>.trace.lisp`)。指定がなければ
  従来どおり各ファイル自身の `<dir>/output/` に書く。
- 対象 `*.lisp` が1件も見つからない場合はエラーとし exit 3(引数ミスの早期検知)。
- 全ファイル実行後、成功/失敗件数と失敗ファイル名の一覧をサマリーとして stderr に出す。
- `--refresh` / `--strict` / `--model` / `--backend` / `--dry-run` / `--no-explore` /
  `--plugin` は全ファイルに一様に適用する(ファイル単位の個別上書きは提供しない)。

### 26. `defspec` / `check-spec` = 仕様の第一級構文(2026-07-20 追加)
- 決定24 の plist 慣習を専用構文に昇格する。`(defspec name . clauses)` は条項を
  **未評価データ**のまま検査して束縛する。値は `:function` を名前で補った条項 plist
  そのもので、従来の plist 慣習(`document-spec` 等の動詞句、`get-property`)と互換。
- スキーマ検査は決定論的(オラクルなし): トップ条項の重複 = `:spec-duplicate-clause`、
  `:invariants` の条項名重複 = `:spec-duplicate-invariant`、`(:name "1文")` 形でない
  不変条件・`:in`/`:out` を欠く実例 = `:spec-malformed-clause`。違反はエラー値(決定13)
  になり、何も束縛しない。
- `def` 系と同様に**束縛した値を返す**(決定17)。result に仕様全文が残り、chain の
  名前復元・`allisp diff` の名前照合(`defspec` を def 系に追加)がそのまま効く。
- アクセサ `spec-invariants` / `spec-invariant` / `spec-examples` は決定論的ビルトイン。
- `(check-spec name [:model M] [:fresh F])` は各不変条件を **1 条項 = 1 オラクル呼び出し**で
  `(lambda (in out) ...)` 述語へ lowering し、全実例へ決定論的に適用する。
  述語のプロンプトには当該条項と `:signature` しか入らない(合成フォームを囲みトップ
  レベルにも使う)ため、**1 条項の編集はその条項の述語キャッシュだけを無効化**する
  (決定4・5 の粒度を条項に細分)。実例の追加・編集は LLM 呼び出しゼロで再検査される。
- 1 組の (in, out) で判定できない条項(冪等性など)はオラクルが intermediate-code を
  返し、`:skipped` に理由付きで記録される(弱い検査への近似を禁止)。違反が 1 件でも
  あれば値は `:spec-violation` のエラー値になり、`:detail` の `(spec-check ...)` が
  どの条項がどの実例で偽かを名指しする。実行主体は常に決定論的評価器(決定19 整合)。
- `defspec` / `check-spec` は予約語。

### 27. `derive` = 派生台帳、`allisp spec status` = 鮮度検査(2026-07-20 追加)
- `(derive <path> :from <spec名|(名...)> :via <生成フォーム>)` は generate-file の
  上位構文。書き出しは `%generate-file` へそのまま委譲する(dry-run、非 .lisp
  ターゲットの文字列規約、provenance、キャッシュ挙動はすべて決定16 のまま)。
- 成功時にプロジェクトルートの **`.allisp/derive.lisp`**(git 管理推奨)へ
  `(derivation :v 1 :target ... :source ... :from ... :from-sha256 ... :via ...
  :target-sha256 ... :generated-at ...)` を target キーで上書き記録する。
  `:from-sha256` は正規化した条項 plist の print の sha256。**defspec の条項が
  未評価データである**(決定26)ため、このハッシュは評価なしにソースの再読で
  再計算でき、status のゼロコスト性が成立する。
- `allisp spec status [<root>]` は台帳を読み、1 target = 1 S 式で
  `(fresh ...)` / `(stale ...)`(ソースの defspec 条項がハッシュ不一致) /
  `(drifted ...)`(target のバイト列が生成時と不一致 = 手編集) /
  `(missing ...)` / `(unknown ...)`(記録ソースに defspec が見つからない等)を出力する。
  exit は全 fresh = 0、それ以外 = 1(diff(1) 規約)。**LLM 呼び出しゼロ**。
- これにより決定24 の「派生物を手で直すな」が規律から機械検査に変わる。stale の解消は
  ソースの再実行(仕様不変ならキャッシュリプレイでバイト同一に戻り fresh 復帰 = $0)。
- `:from` の各名前は評価環境で仕様(リスト値)に束縛されていなければエラー値。
  status の鮮度判定は記録された `:source` ファイル内の defspec に限る(他ファイル
  由来の仕様は `unknown` と報告する)。`derive` は予約語。

### 28. `probe-spec` = 仕様の穴の能動探索(2026-07-20 追加)
- `(probe-spec <spec> [:focus (条項名...)] [:model M] [:fresh F])` は決定24 の
  query-spec 慣習(受動・単一質問)を一般化した特殊形式: オラクルに条項間の矛盾と
  未規定の入力領域を**能動的に**探索させる。
- spec 指定子は pure 相当で決定論的に評価する(未束縛は即エラー値、オラクル浪費なし。
  決定23 の入力評価と同じ)。
- 応答契約は「`(spec-findings :spec <名> :count <n> :findings (...))` 1 式」(決定6 の枠内)。
  各 finding は intermediate-code(決定19 の不活性データ): `:why` が矛盾・沈黙する
  条項名を、`:how` が決着させる `:examples` 条項を指す。穴がなければ `:findings ()`。
- 生成物は**実行しない**(`oracle-eval :execute nil`、決定23 と同機構、trace は
  `:status :generated`)。評価器は形を検査して findings を正規化し、不適合応答は
  `:invalid-probe-response` のエラー値にする。
- キャッシュ粒度は**意図的に仕様全体**(仕様は文脈同梱でプロンプトに入る): 穴は条項の
  組に宿るため、どの条項の編集も再探索を要する。check-spec の条項単位キャッシュとの
  対比を docs に明記する。`:focus` はプロンプトの監査節に入るため自然に別キーになる。
- `probe-spec` は予約語。

### 29. `verify` = 外部テスト実行ゲート(2026-07-20 追加)
- `(verify <コマンド文字列> [:targets (パス...)] [:expect N])` は**不活性な登録フォーム**。
  評価時には検証レコード `(verification ... :status :pending)` を返して run に登録する
  だけで、外部コードを一切実行しない(決定19「実行主体は Lisp か外部ツール」を維持)。
- 実行は CLI 層の明示フラグ **`allisp run --verify`** の下でのみ: 全トップレベル
  フォームの評価と全ファイル生成の**後**、result 書き出しの前に、登録順に
  `sh -c <コマンド>`(cwd = ソースファイルのディレクトリ)で実行する。LLM は非関与。
- exit が `:expect`(既定 0)と一致すればレコードは `:status :passed` に完成し、
  不一致なら **result 上のそのフォームの値が `:verification-failed` のエラー値**
  (`:detail` に exit・stderr 末尾)に書き換わる。決定13 の部分評価・exit 規約と
  `allisp diff` の感度分析にそのまま乗る。`--strict` では最初の失敗で残りを
  `:skipped` にして打ち切る。dry-run は実行せず予告のみ。
- 成功時、`:targets` のうち台帳(決定27)に載っており**生成時とバイト同一**の
  ファイルへ `:verified-at` を刻む。`spec status` は `(fresh ... :verified t)` と
  報告する(編集済みファイルの検証は仕様について何も証明しないため刻まない)。
- `verify` は予約語。

## 実装で確定した追加セマンティクス(2026-07-15 実装時)

- **Lisp-1**: 関数と変数の名前空間は単一。`(mapcar f list)` が自然に書ける。
- **正規順序オラクル**: 未束縛フォームは**引数を評価せず**フォーム全体をオラクルへ渡す。
  `(calculation (- 40 :work-hours))` のようなキーワード混じりの抽象式が決定論的評価器で壊れない。
  引数の値が必要な場合は環境束縛として文脈に同梱される(決定5と整合)。
- **効果位置エスカレーション**: 値が捨てられる位置(let/progn の非末尾など)の未束縛フォームは、
  loweringに必要な全体の意味を保つため、**最も近い値位置の囲みフォームごと**オラクルに送る。
  例: `(let ((x 21)) (mystery x) (* x 2))` は let 全体が1回のオラクル呼び出しになる。
- **`def` の多値形**: `(def name expr)` は評価、`(def name f1 f2 ...)` はフォーム列を未評価データとして束縛
  (散文的な定数表を記述するため)。
- **`def` 系の評価値 = 束縛した値**(2026-07-16 変更、決定17 の前提): CL と異なり名前ではなく値を返す。
  runner が :value にそのまま値を記録でき、記録と実態が常に一致する。defun / defmacro は名前を返す。
- **T/NIL**: `CL:T`/`CL:NIL` のみ user パッケージに import。他の CL シンボルは一切見えない。
- **readtable :invert**: 小文字ソースがそのまま小文字で往復する。quote/backquote/comma は
  独自リーダーマクロで `(quote ...)`/`(quasiquote ...)`/`(unquote ...)` の平リストに読む。
  全角スペース(U+3000)は空白として読む(2026-07-18 追加。日本語ソースで
  `:keyword` の前の全角スペースがパッケージ名に化けるのを防ぐ)。
- **ビルトイン追加**: DSL 頻出の `filter` / `equal?` / `get-property`(シンボル名一致の plist 引き)を
  決定論的ビルトインに含めた。
- **マクロの &key**: 奇数個・非キーワード混在の引数列でもエラーにせず黙って既定値
  (思考 DSL はキーワードとフォームを自由に混ぜるため)。
- **キャッシュキー**: sha256(プロンプト版 + モデル + 完成プロンプト全文)。文脈・囲みトップレベル・対象フォームはプロンプトに含まれる。
